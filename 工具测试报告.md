## 二、问题清单（按严重程度排序）

### 🔴 严重问题（3个——影响核心工作流）

#### 1. MCP filesystem 服务器配置缺失 —— 14 个工具全部不可用

**涉及工具**：`mcp__filesystem__read_file`, `read_text_file`, `read_media_file`, `read_multiple_files`, `write_file`, `edit_file`, `create_directory`, `list_directory`, `list_directory_with_sizes`, `directory_tree`, `move_file`, `search_files`, `get_file_info`, `list_allowed_directories`

**错误表现**：
```
list_allowed_directories → 返回空列表
get_file_info(F:\syproject\gou-ide\go.mod) → "Access denied - path outside allowed directories"
```

**根本原因**：MCP filesystem 服务启动时没有传入 `allowed-directories` 参数，导致没有授权任何目录。配置中的命令仅为：
```
npx -y @modelcontextprotocol/server-filesystem
```
缺少应有的 `F:\syproject\gou-ide` 参数。

**修复建议**：重新配置 MCP filesystem，在 `mcp_configure` 中将参数改为：
```json
{
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-filesystem", "F:\\syproject\\gou-ide"]
}
```

#### 2. CGO 依赖导致 go_build / go_run / run_test 编译失败

**涉及工具**：`go_build`, `go_run`, `run_test`, `debug_start`

**错误表现**：
```
go_build ./cmd/companion/ → 
# github.com/hoonfeng/goui/pkg/canvas
pkg\canvas\skiacanvas.go:30:39: undefined: goskia.Typeface
...
FAIL  github.com/hoonfeng/paircode/cmd/companion [build failed]
```

**根本原因**：项目依赖 `github.com/hoonfeng/goskia` 包，该包使用 `import "C"`（CGO），必须设置 `CGO_ENABLED=1` 并安装 C 编译器（MinGW-w64）才能编译。`go_build`/`go_run`/`run_test`/`debug_start` 等工具在执行 Go 命令时没有自动设置 `CGO_ENABLED=1`。

**修复建议**：
- **短期**：调用这些工具前先用 `run_command` 设置环境变量再执行
- **长期**：在 `go_build`/`go_run`/`run_test` 后端解析项目的 `CLAUDE.md` 或检测 `import "C"` 后自动设置 CGO_ENABLED=1
- 或通过 `bridge_takeover` → `bridge_register_system_tool` 创建一个自定义构建工具

### ⚠️ 中度问题（3个——需要留意但可规避）

#### 3. web_search DuckDuckGo 超时

**涉及工具**：`web_search`

**错误表现**：
```
web_search("Go 1.24 release date") → 
Error: Get "https://html.duckduckgo.com/...": context deadline exceeded
```

**根本原因**：SearXNG 未配置（搜索结果为空），fallback 到 DuckDuckGo HTML 接口，但该接口在国内网络环境不可达（超时 30s）。

**修复建议**：
- 部署自建 SearXNG 实例
- 或使用 `web_fetch` 调用其他搜索引擎 API（如 Bing Search API）
- 临时规避：直接用 `web_fetch("https://www.google.com/search?q=<query>")` 替代

---

#### 4. headless_browser 默认模式返回空内容

**涉及工具**：`headless_browser`

**错误表现**：
```
headless_browser("https://example.com") → {"text":"","charCount":0}
headless_browser("https://example.com", raw=true) → {"text":"Example Domain...","charCount":127}
```

**根本原因**：默认的文本提取模式（非 raw）在处理标准 HTML 页面时提取逻辑出现问题，提取的 visible text 为空。`raw=true` 模式使用不同的文本提取策略，正常工作。

**修复建议**：
- 调用时始终传 `raw=true` 参数
- 或修改 `cmd/companion/agent/headless.go` 中的默认文本提取逻辑

---

#### 5. find_symbol_usages 对某些符号查找失败

**涉及工具**：`find_symbol_usages`

**错误表现**：
```
find_symbol_usages(cmd/companion/main.go, "main") → 
"在 cmd/companion/main.go 中未找到任何符号，无法定位符号 'main'"
```

**根本原因**：该工具内部先调用 `get_file_symbols` 在文件中查找符号，再通过 LSP 查引用。`get_file_symbols` 通过 AST 扫描可能对某些符号（如顶层的 `func main`）匹配有歧义。

**修复建议**：调用时配合 `lsp_references` 工具直接使用 LSP 查询（已验证可正常工作），或精确指定行号：
```
lsp_references(file=cmd/companion/main.go, line=66, symbol="main") → ✅ 返回 1 个引用
```

### ℹ️ 低度问题 / 信息提示（3个）

#### 6. find_config_files 未排除 .Pair/ 等内部目录

**涉及工具**：`find_config_files`

**问题描述**：返回结果中包含大量 `.Pair/plans/` 和 `.Pair/conversations/` 下的 JSON 文件，这些并非传统意义上的"配置文件"。

**修复建议**：增加排除列表，过滤掉 `.Pair/`、`node_modules/`、`vendor/` 等目录。

#### 7. find_symbol / get_file_symbols 对非 .go 文件报错

**涉及工具**：`find_symbol`, `get_file_symbols`

**问题描述**：对 `go.mod` 等非 `.go` 后缀文件报错"不是 Go 源文件"。

**评价**：这是预期行为，文档已写明仅支持 Go 源文件。

#### 8. go_build 传入不存在的路径

**涉及工具**：`go_build`

**问题描述**：`go_build(path="cmd/checkgoskia")` 报错"包目录不可访问"——该目录不存在。

**评价**：这是使用方问题，非工具本身bug。调用前应先 `list_files` 确认路径存在。
