# 伴随式 CodeAgent · Go 重构开发规则（防漂移）

> 本文件是这个重构项目的**唯一事实来源**。任何开发动作开始前先对照本文件；与本文件冲突的做法一律以本文件为准。新增的硬性约定也写回这里。

## 1. 项目定位

- **目标**：用 [goui](../../README.md)（Go 自绘 UI 库，Skia GPU 渲染）**全 Go 重写**「伴随式 CodeAgent」——原项目是 Electron + React 19 + TypeScript 的 IDE 式 AI 编码助手。
- **落地位置**：goui 仓内 `cmd/companion/`，直接 `import` 本地 goui 包（`internal/widget` 等）。改库与写应用在同一仓，迭代最快。
- **参考源（只读）**：`F:\syproject\伴随式codeagent`。重构时读它理解原有 UI/逻辑/交互，**绝不修改它**。
- **范围**：UI + Agent 引擎（TAOR 主循环 / 多 LLM 适配 / MCP / 规划系统）**都用 Go 重写**。不引入 Electron / Node / TS。

## 2. 铁律（违反即返工）

1. **图标只用 Lucide，严禁 emoji / Unicode 字符当图标。**
   - 一律 `widget.Lucide("icon-name")`（向量 path 渲染，统一线性风格）。
   - ✕ ☐ 🪟 ▲ ▶ — ☰ ⚙ 🔍 📁 ✓ ⚠ 这类字符**禁止**出现在任何 UI 文案/按钮/标签作为图标。
   - 缺图标 → 去 Lucide 补 SVG path 数据进图标表，**不准用字符替代**。
2. **组件使用要"代码最少 + 定义最方便"。** 一行能写出来的组件不写三行；优先声明式 + 链式 setter + 友好默认值。若某组件用起来啰嗦，**改库让它更好用**。
3. **UI 出 bug → 修 goui 库本身**（`internal/widget`/`canvas`/`layout`…），不在应用层绕过/打补丁。
4. **组件不满足 → 改/加 goui 组件**，让库更完整通用，不在应用层硬凑一次性代码。
5. **重构中发现不合理 → 顺手优化**（库或应用），但优化要可验证、不破坏既有测试。
6. **全 Go**：Agent 引擎也是 Go；不依赖原 TS 实现运行。
7. **样式 vs 能力的边界**：样式（粗细/颜色/间距）走**参数**（组件字段 + 链式 setter，由调用方传；应用侧集中在自己的样式 helper 里）；能力/行为（hover、拖动、填充不留白等）做进**组件本体**；UI bug 修组件本体。**别为了调样式去改组件的默认值或实现**。

## 4. 图标系统（Lucide）

- 入口：`widget.Lucide(name string, opts ...IconOption)` —— 返回一个图标 Widget，默认 24px、`currentColor`（跟随主题文字色）、stroke 风格。
- 实现：Lucide SVG `path d=...` → `svg.path.ParsePath` 解析为命令 → 直接发射 `canvas.Path` **原生曲线**（`CubicTo`/`QuadTo`/arc→cubic，**不展平折线**）→ `Canvas.DrawPath`（stroke）。向量、可缩放、可着色、GPU 渲染，**不走光栅化**。
- 数据：`internal/widget/lucide.go` 内 `lucideIcons` 表（图标名 → 一组 SVG path d）。按需增补，来源取 Lucide 官方（MIT）。复杂图标（含曲线/弧线）现已正确渲染。

## 5. 状态管理范式

- 不照搬 Zustand。用 **Go 结构体 + goui 的 SetState/观察者**：状态结构体持有数据，变更走 `SetState`（触发 relayout/repaint，见 [[goui-setstate-deep-update]]）。
- 跨面板共享状态放 `state/`，面板订阅自己关心的切片。

## 6. 验证纪律

- 每做一个面板/组件：① headless 渲染 PNG 自检布局/视觉 ② 实机 `go run ./cmd/companion/` 跑一遍交互。
- 改了 goui 库：跑 `go test ./internal/...`，确保不回归。
- 窗口/交互类无法 headless 的，构建 exe 实机验证 + 看日志（退出 exit 0、无报错）。

### 新增的 goui 组件（改库所得，通用）
- **`ResizeHandle`** `internal/widget/resizehandle.go`：可拖动分隔条，兼作分隔线。`widget.VResize(line, active, onDrag)`/`HResize(...)`；回调位移增量 `delta`（符号由上层定）。**样式=参数**：`Line`/`Active` 颜色 + 链式 `.WithThickness(n)`/`.WithColors(line,active)`。**能力=组件本体**：平时填 `Line`、hover/拖动填 `Active` 高亮、整条填充（不留白）。companion 把样式集中在 `vDivide`/`hDivide` helper（粗细 `dividerW=4` + 色），改样式不碰库。
- **拖动中断修复**（拖动「几像素就停、鼠标脱离分隔条」的真凶）：`ResizeHandle`/`Splitter` 的 `HandleEvent` 此前在 `MouseLeave` 清 `dragging`。拖动中鼠标一旦移出这条 6px 窄条，app 的 hover 检测就给它派发 `MouseLeave` → 拖动半途中断。修复：**只 MouseUp 结束拖动**（已 capture，move/up 必达本元素，无需 Leave 兜底）。复现测试 `internal/app/resizedrag_test.go`（拖动中 move 出 handle，断言尺寸仍变）。⚠️ 教训：headless 复现拖动 bug，鼠标坐标必须移出元素，否则 hover 不变、测不出。
- **鼠标捕获修复**（win32，拖动/点击地基）：`WM_LBUTTONDOWN`→`SetCapture` / `WM_LBUTTONUP`→`ReleaseCapture`。此前缺失 → 拖动掠过非客户区/快速移动就脱钩、靠窗口边的按钮 up 丢失致 click 不触发。与上条配套，拖动/点击双保险。
- **按钮内图标拦截点击修复**（「标题栏按钮全不工作」真凶）：`HitTest` 返回最内层命中 = 按钮里的 `lucideIconElement`，捕获落在图标上，`MouseUp` 直接发图标、不冒泡 → Button 永远收不到 up、`OnClick` 不触发。修复：装饰型元素实现 `widget.HitTestIgnorer.IgnoreHitTest()=true`，`pipeline.hitTestRecursive` 命中穿透到可交互祖先（Button）。Lucide 图标已接。这是 Button 支持 child 图标后才暴露的连带 bug。
- **捕获路径 MouseUp 必须冒泡**（`internal/app/app.go` `routeMouseEvent`，「按钮里放『图标+文字』容器→点击没反应」真凶，与上条同源更彻底）：MouseDown 把**最深命中元素**设为 `capturedElement`；当 child 是单图标时靠 `HitTestIgnorer` 穿透使捕获元素=Button 才有效，但 child 是 `Div(图标+文字)` 这种**容器**时，捕获的是内层 Div/Text，旧捕获路径 MouseUp **只发捕获元素本身、不冒泡** → 祖先 Button 收不到 up、`OnClick` 不触发。修复：捕获路径的 down/up 改为**沿父链冒泡**（与非捕获路径一致，尊重 `StopPropagation`）。不影响拖动（拖动走更前面的 `isDragging` 分支）。回归测试 `internal/app/buttonclick_test.go`（点击落在按钮内层容器，断言 OnClick 触发 1 次；去掉修复则 0 次）。⚠️ 凡 `Button{Child: Div(...)}`（带文字的 pill、标签按钮）都吃这条。
- **Button 有 child 时按内容紧凑**（`internal/widget/basic.go` Layout，「文字图标看着没居中」真凶）：`minW` 的 64 默认**只给文本按钮**（`e.child==nil`）；有自定义子内容时不强加 64，按 `内容+padding` 紧凑。否则小内容（如「自动」胶囊 ~36px）被撑到 81px、内容居中后两侧浮空 ~22px，看着像没居中。需要最小宽自设 `MinWidth`（图标按钮 toolBtn=26/iconGhost=24 已显式设，不受影响）。
  - ⚠️ **同病的「垂直」版（半个真相）**：`minH` 仍默认 32；小按钮（图标+文字 child）不设 `MinHeight` 会被撑到 32px 高、内容浮空。修法：自设小 `MinHeight` 哨兵 + 文字用 `tightText`（`LineHeight`=字高，免 1.3× 行高撑高文字盒）。**但这只是一半**——把盒子缩贴合后用户仍说没居中，真凶是 `BaselineFor` 文字基线偏高（见上「文字垂直居中偏高修正」）。两者叠加才真居中：盒子贴合内容 + 基线修正。**口诀升级：内容浮在过大盒子里＝看着没居中（缩盒子解）；但若盒子已贴合还偏，那是文字基线本身偏高（修 BaselineFor）。**
- **Input/Textarea 三态边框都要设**（`internal/widget/input.go`，「输入框悬停冒灰边」真凶）：`HoverBorderColor` 不设时默认 el 调色板 `#c0c4cc` 浅灰；想无边框（边框由外层包裹盒提供）须把 `BorderColor`/`FocusBorderColor`/**`HoverBorderColor`** 三者都设为底色。只设前两个 → 悬停照样冒一圈灰边。
- **诊断神器（无需用户）**：`Add-Type` 调 Win32 `EnumWindows`(按 pid 找窗口)+`GetWindowRect`+`SetCursorPos`+`mouse_event`(合成真实 down/up 点击)，配合临时 `log.Printf` 抓 `WM_NCHITTEST`/`route DOWN hit=%T`/`OnClick`，自动定位实机交互 bug。实测：close→exit0、toggle→OnClick。
- **鼠标光标系统**：Element 实现 `widget.CursorProvider`（`Cursor() int` 返回 `widget.Cursor*` 形状）即可在悬停/拖动时自动切换光标。链路：win32(`WM_SETCURSOR`+`SetCursor`)→`window.SetCursor`→`widget.SetCursorShape`→`app.applyCursor`（按 捕获/悬停 元素）。ResizeHandle 已接（竖条 ↔ / 横条 ↕）。
- **图标 HitTest 穿透** `widget.HitTestIgnorer`：装饰型元素（图标）实现 `IgnoreHitTest()=true`，`pipeline.hitTestRecursive` 穿透到可交互祖先。Lucide 图标已接（否则按钮内图标截走点击）。
- **Input 受控清空** `internal/widget/input.go`：`Input.ResetToken int`，改变它(+SetState) → `InputElement.Update` 把运行时文本重置为 `Text`（发送后清空输入框等；goui Input 默认非受控）。
- **ScrollView 受控滚到底** `internal/widget/scroll.go`：`ScrollView.ScrollEndToken int`，改变它(+SetState) → 下次布局跳到底部（聊天追加消息后保持看见最新）。
- **Flex 拉伸列多行文本不裁** `internal/widget/flex.go`：cross-stretch 列第一遍即按最终交叉宽(tight)测量子，使多行文本据此算对行高；否则第二遍只拉宽、锁死第一遍高度 → 行数变多被裁。
- **文本换行算法统一** `internal/widget/basic.go`：`TextElement` 的 Layout 测高与 Paint 渲染共用 `splitLines`（空格单词换行）；此前 Layout 按字符二分、Paint 按空格换行，中英混排行数不一致 → 裁字。
- **字体度量改用 Skia 真实值（删私有经验估算）** —— 文字垂直居中偏高的**根治**。原 `internal/canvas/fontutil.go` 的 `GetFaceAscent/Descent/LineHeight` 是私有经验值 `size×0.8 / 0.2 / 1.3`，且 `SkiaCanvas.MeasureText` 也拿 `size×0.8/0.2` 充数——全是拍脑袋、与真实字体不符（阿里普惠体实测 ascent≈1.06×/descent≈0.34×，经验值差一大截）→ 基线算偏、文字上飘。改法：**全部走 Skia 的 `sk_font_get_metrics`**。① goskia 新增 `Font.Metrics()`（绑定 `sk_font_get_metrics`，返回 ascent/descent/leading/capHeight…，见 `F:\syproject\goskia\skia\text.go`）；② goui `SkiaCanvas.fontVMetrics(font)` 取真实度量，`MeasureText` 用它填 Ascent/Descent；③ `GetFaceAscent/Descent/LineHeight` 改为查 Skia（按字号缓存 `faceVMetrics`）；④ 删掉上一版的 `vCenterFix` 补丁。影响全库居中文字。⚠️ 教训：**用 Skia 渲染就别再自己估字体度量**。（注：真实度量让基线大致对，但**中文仍偏上**——见下条「墨迹居中」才彻底修。）
- **文字按实际墨迹垂直居中（中文偏上的彻底修）** —— 即便用真实 ascent/descent，中文方块字**底边贴基线、几乎不占字体 descent**（descent 是为拉丁下伸部 g/p/y 预留的），用 `(ascent-descent)/2` 居中会把中文顶高 ~0.08×字号、与同行**几何居中的图标错位**（文字比图标高一截，用户截图实锤）。根治：**用文字的实际墨迹包围盒（Skia `sk_font_measure_text` 的 tight bounds）居中**，而非字体度量。① goskia `Font.MeasureText` 本就返回 bounds；② goui `TextMetrics` 加 `InkTop/InkBottom`（相对基线），`SkiaCanvas.MeasureText` 取各 run 的 min-top/max-bottom 填上；③ `textdraw.go` 新增 `BaselineForMiddle(c,text,font,top,h)`：`baseline = 盒中线 − 墨迹中心`，`DrawTextAligned` 的 `VAlignMiddle` 与 `TextElement.Paint`（逐行）都改用它（拿不到墨迹回退 `BaselineFor`）。影响全库居中文字（Button/Tag/Badge/Input…），skiawidget 全组件+主窗壳实拍无回归、中文与图标真正对齐。⚠️ 教训：**方块字居中别信字体 ascent/descent，要按实际墨迹（tight bounds）**。
- **墨迹度量必须走共享(raster)画布，不用渲染画布**（`textdraw.go` `BaselineForMiddle`，「headless 截图居中、实机 GPU 窗口偏」真凶）：原 `BaselineForMiddle` 用传入的渲染画布 `c.MeasureText` 取墨迹盒。GPU 渲染画布在 Paint 期（GL 上下文激活、Skia flush 中）`MeasureText` **取不到墨迹盒（返回空）** → 回退成 ascent/descent 居中（偏）。改：墨迹度量统一走 `MeasureTextGlobal`（共享 raster 测量画布，与字号行高同源），与渲染后端无关 → headless/GPU 一致。**诊断法**：`Start-Process` 启 exe + Win32 `PrintWindow(flag 2)` 抓真实 GPU 窗口 → 逐行扫像素量「内容上/下留白」定方向（眼睛会骗人，像素不会）。
- **Button 有 child 时必须按内容自然高布局（child 偏下真凶）** `internal/widget/basic.go` Layout：原把按钮可用高（`ctx.MaxHeight`）传给 child 测量。child 是「图标+gap占位Div+文字」flex 行时，**0 高的 gap 占位 Div 会拉伸撑满该可用高** → 行高被顶到 24（=同排 paperclip 的 MinHeight）→ 按钮随之变高、child 被 `paddingTop` 顶偏 → 内容**整体偏下** ~paddingTop（用户实机截图+PrintWindow 像素实测 top/bot 留白 9/4 锤实）。修：child 布局传 **`MaxHeight: 1<<30`（高度不限）**，让 child 取自然高，按钮 = 内容+padding、再居中。修后窗口实测 top/bot 留白 5/6（居中）。⚠️ 教训：容器给子传「可用尺寸」当约束，会让 flex/占位子**拉伸**；要子取自然尺寸就放开该轴约束。
- **⚠️ `HBox`/`VBox` 默认 CrossStart（顶/左对齐），不是居中**（`flex.go` `NewRow`/`NewColumn` 不设 `CrossAxisAlignment`，零值=CrossStart；不同于 Flutter 的 center）。后果：`Div(Height:X){ HBox(...) }` 这种**定高条里塞 HBox**，内容会被顶到条顶——标题栏「伴随式 CodeAgent」、面板头「文件/终端」都中招（顶对齐、没垂直居中）。**修法/铁律：定高的横条别用 `Div(Height,{HBox})`，直接写 `Div(Style{Height:X, FlexDirection:"row", AlignItems:"center"}, 图标, 文字…)`**（容器自身即居中行）。companion 的 `titleBar`/`panelHeader` 已改；`statusBar`/`statusItem`/聊天面板各行本就显式 `AlignItems:center`，无碍。排查范围：定高/定宽容器内的单行内容。
